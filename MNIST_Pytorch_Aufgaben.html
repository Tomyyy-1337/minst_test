Tipps und Hinweise:

Es knnte ntzlich sein, dem Konstruktor zwei weitere Parameter zu bergeben, z.B. root_dir und loader_mode. root_dir wre der Wert aus der Zelle zuvor und kann im Konstruktor genutzt werden um die Pfade der Bilder in einer Liste oder einem Array zu speichern. loader_mode knnte ein String wie z.B.   2. Im Allgemeinen werden im Konstruktor lediglich die Pfade, jedoch nicht die Bilder selbst geladen. Das liegt daran, dass moderne Datenstze sehr gro sind. Wenn Sie dann alle Bilder laden wrden, kann es schnell passieren, dass Ihr Arbeitsspeicher berluft.
  3. Angenommen Ihr Dataloader soll die Trainingsbilder laden. Wenn Sie im Konstruktor eine Liste oder Array, aller Trainingsbilder erstellen, dann muss __len__(self) lediglich die Lnge dieser Liste zurckgeben.

  5. Beachten Sie, dass Sie das geladene Bild (je nach verwendetem Package ein numpy-Array oder ein anderes Objekt) in ein Pytorch Tensor konvertieren mssen.
  6. Denken Sie daran, dass __getitem__(self, idx) nicht nur das Bild, sondern auch das Label des Bildes zurckgeben muss.

import glob
import torch
import torchvision
import cv2

from torch.utils.data import Dataset


class Simple_Dataloader(Dataset):
    def __init__(self,root_dir,loader_mode):
        self.root_dir = root_dir
        self.loader_mode = loader_mode

        self.data = torchvision.datasets.ImageFolder(self.root_dir + self.loader_mode)
        
        self.transform = torchvision.transforms.Compose([
            torchvision.transforms.ToTensor()
        ])

    def __len__(self):
        return len(self.data)

    def __getitem__(self, idx):
        img, lbl = self.data[idx]
        # Transform Numpy Array into Pytorch Tensor
        img_torch = self.transform(img)
        
        # Flatten Tensor
        img_flattened = torch.flatten(img_torch)

        return img_flattened, lbl

Zuvor haben wir eine Subclass der Klasse Dataset erstellt. Damit ist klar, wie die Daten geladen und vorbereitet werden. In modernen Frameworks wie Pytorch, Tensorflow und Keras werden einzelne Sampels (in unserem Fall also Bilder) zu einer so genannten Batch gruppiert. Die Gre der Batch hngt in der Regel vom vorhandenen VRAM oder im Falle eines Trainings mit CPU vom Arbeitsspeicher ab. Oft werden Batchgren zwischen 8 und 128 verwendet. Bei einer Batch von 1, redet man oft auch von einer Mini-Batch. Das Trainieren mit einer Mini-Batch ist selten effektiv, da die Gewichte nach jedem Sample angepasst werden und dies die Konvergenz behindern kann. Wre die Batch so gro wie der ganze Datensatz, dann wre zwar das Modell perfekt auf unsere Trainingsdaten angepasst, aber mglicherweise die generallisierung nicht so gut, wie wenn eine kleinere Batch gewhlt worden wre. Daher whlt man auch bei sehr guter Hardware, selten eine Batch die grer ist als 512. Da die MNIST Bilder sehr klein sind, sollte eine Batch zwischen 32 und 128 machbar sein.

4. Definition des Modells (ToDo)

Jetzt sind wir bereit unser erstes MLP zu definieren. Betrachten wir hierzu folgende Modell-Vorlage:

input_layer = nn.Linear(784, 128)
hidden1 = nn.Linear(128, 64)
output_layer = nn.Linear(64, 10)

In dieser Aufgabe betrachten wir nur Fully-Connected Layers. Das sind Layer, in denen jedes Neuron des i-ten Layers mit jedem Neuron des (i+1)-ten Layers verbunden sind. In Pytorch verwendet man hierfr nn.Linear(in_features, out_features). Da all unsere 28 x 28 Bilder zuvor zu einem Vektor mit 784 Eintrgen konvertiert wurden, ist in_features im input_layer 784. Eine andere Zahl, die sich in dieser Aufgabe ebenfalls nicht ndert, ist die Anzahl der Klassen: Wir betrachten ausschlielich Bilder die Zahlen zwischen 0 und 9 zeigen. Daher hat der Parameter out_features im Layer output_layer den Wert 10. Fr alle andere Zahlen gilt: Die Anzahl der out_features im Layer i, muss exakt mit der Anzahl der in_features im Layer i+1 bereinstimmen.

Als nchstes betrachten wir die forward(self, x) Methode.

def forward(self, x):
    x1 = torch.nn.functional.relu(self.input_layer(x))
    x2 = torch.nn.functional.relu(self.hidden1(x1))
    x_out = torch.nn.functional.relu(self.output_layer(x2))
    return x_out

Bisher ist noch nicht klar in welcher Reihenfolge die zuvor definierten Layer angewendet werden und wir haben uns noch keine Gedanken um die Aktivierungsfunktionen gemacht. Wenn das Modell auf eine Eingabe angewendet wird, dann wird diese Methode aufgerufen, dabei ist x die Eingabe, also unser Vektor mit 784 Eintrgen. In diesem Beispiel wird die Eingabe x zunchst auf den Layer input_layer angewendet und auf jedes der out_features = 128 Neuronen wird die Aktivierungsfunktion ReLu angewendet. Dieses Zwischenergebnis speichern wir in der Variable x1. Nun nehmen wir dieses Zwischenergebnis (ein Vektor mit 128 Eintrgen) und wenden den Layer hidden1 darauf an und anschlieend wenden wir wieder die Aktivierungsfunktion ReLu an. Das gleiche geschieht jetzt noch ein drittes Mal, bevor x_out (also ein Vektor mit 10 Eintrgen) zurckgegeben wird. Da wir auch beim dritten Mal ReLu auf alle Ausgabeneuronen angewendet haben, liegt jeder Wert in x_out im Bereich $[0, \infty[$

Pro-Tipp (nicht prfungsrelevant): Ein Problem bei SGD ist, dass die Lernrate eine "globale Lernrate" ist und somit fr alle Richtungen gleich bleibt. Bei mehrdimensionalen Funktionen kann das aber zum Problem werden. Denn wenn Ihre aktuellen Gewichte eine Position auf der mehrdimensionalen Funktion beschreiben, die in einer Dimension ein Plateau ist und in einer anderen Richtung eine sehr starke Steigung hat, dann wrde SGD in der Richtung des Plateaus sich kaum bewegen und in der Richtung der starken Steigung einen sehr groen Sprung machen. Um das zu verhindern fhrt Adam eine adaptive Lernrate fr jede Richtung/Dimension ein ein. Diese sorgt dafr, dass in Richtungen eines Plateaus grere Sprnge gemacht werden als in Richtungen mit starken Steigungen. Adam hat sich daher als sehr effektiv herausgestellt und konvergiert viel schneller als SGD. Das ist allerdings keine Garantie dafr, dass das Minimum in dem Adam landet auch stets besser ist als das mit SGD.

6. Training und Validierung implementieren (ToDo)

Jetzt sollen Sie das Training und die Validierung implementieren. Um den Aufbau besser erklren zu knnen fhren wir zunchst drei neue Wrter ein:

  1. Epoche: Eine Epoche beschreibt eine vollstndige Trainings- und Validierungsphase.
  2. Phase: Es gibt die Trainings- und die Validierungsphase. Die Validierungsphase folgt immer nach der Trainingsphase.
  3. Iteration: Zuvor haben wir schon ber eine Batch geredet. Der Dataloader ld also eine Batch und die zugehrigen Labels. Bei einer Datensatzgre von $n$ Samples, entstehen $\big \lfloor\frac{n}{\text{batch size}} \big \rfloor$ Batches. In jeder Phase iterieren wir ber alle Batches. Einen dieser Schleifendurchlufe nennen wir Iteration.

Bei der Implementierung knnen sich an folgender Struktur orientieren:

  1. Zuncht bentigen wir eine Schleife die ber alle Epochen luft, also von 1 bis num_epochs.
  2. Die Phasen knnen direkt hintereinander implementiert werden oder als Schleife mit zwei Durchlufen, einmal fr die Trainingsphase und einmal fr die Validierungsphase. Das hintereinander Schreiben ist fr viele (Anfnger) intuitiver. 2.1. In der Trainingsphase setzen wir das Modell auf trainieren und verwenden hierfr: model.train()
    2.2. In der Validierungsphase setzen wir das Modell auf evaluieren und verwenden hierfr: model.eval()
  3. Fr die Iterationen bentigen wir wieder eine Schleife, z.B. von der Form: for inputs, labels in train_dataloader:
    3.1. Wenn Sie fr das Training eine Grafikkarte verwenden und zuvor das Modell in die Grafikkarte geladen haben, dann mssen Sie auch ihre Eingaben und label auf die Grafikkarte laden. Das knnen Sie so machen: inputs = inputs.to(device)
  4. Gewichte werden nur in der Trainingsphase angepasst, nicht in der Valdierungsphase.
  5. In jeder Iteration sollten Sie mit optimizer.zero_grad() die Gradienten zurck auf 0 setzen. Ohne zu sehr ins Detail zu gehen: optimizer.zero_grad() muss nicht zwingend in jeder Iteration zurckgesetzt werden, ist fr uns aber einfacher.
  6. Um das Training nicht unntig zu verlangsamen, sollten wir Pytorch mitteilen, wann wir uns in der Trainingsphase befinden. Dies knnen wir mit
    with torch.set_grad_enabled(phase == 'train'):
    machen. Dadurch wei Pytorch ob im Loss die Graidenten berechnet werden mssen oder nicht. In der Validierungsphase werden wir die Gewichte nicht anpassen, daher bentigen wir auch keine Gradientenberechnung.
  7. Direkt danach, knnen wir mittles outputs = model(inputs) die Modellausgabe fr unsere Batch inputs berechnen.
  8. Anschlieend berechnen wir den Fehler zwischen der Modellausgabe und der erwarteten Ausgabe (dem Ground Truth): loss = criterion(outputs, labels).
  9. Bevor wir den with torch.set_grad_enabled(phase == 'train'): Block verlassen, mssen wir mittels loss.backward() den Fehler zu allen Gewichten zurckpropagieren und anschlieend mit Hilfe von optimizer.step() die Gewichte aktualisieren.
  10. Damit wir erkennen ob unser Modell overfitted, berechnen wir ein Trainingsloss und ein Validierungsloss.

from sklearn.metrics import confusion_matrix
def calculate_metrics_from_confusion_matrix(conf_matrix):
    # Calculate True Positives, False Positives, False Negatives
    TP = conf_matrix.diagonal() #Calculate True Positives
    FP = conf_matrix.sum(axis=0) - TP #Calculate False Positives
    FN = conf_matrix.sum(axis=1) - TP #Calculate False Positives

    #Precision
    precision = TP / (TP + FP)
    precision = precision.mean()

    #Recall
    recall = TP / (TP + FN)
    recall = recall.mean()

    #F1-score
    f1 = 2 * (precision * recall) / (precision + recall)

    return precision, recall, f1
